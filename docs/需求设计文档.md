## 1. 产品愿景与问题定义

### 1.1 要解决什么问题

有创伤经历、正在自我疗愈的成年人，往往在关系、身体与自我价值上反复受挫。他们需要的不只是「聊天机器人」，而是能够：

- **被坚定地理解与认可**，而不是被评判为「太脆弱 / 还没好」；
- **把困扰说透、看清**，看见模式与成长；
- 在长期使用中被记住（画像、摘要、重要事件），而不是每轮从零开始。

### 1.2 目标用户

| 维度 | 说明 |
|------|------|
| 主用户 | 有创伤经历、正在自我疗愈的成年人 |
| 边界 | **宽于诊断标签**：包括尚未达到 CPTSD 等诊断标准、但同样感到痛苦的幸存者 |
| 非目标 | 危机热线替代、精神科处方、面对面治疗替代；不承诺临床诊断 |

### 1.3 产品角色

| 角色 | 使用者 | 客户端 |
|------|--------|--------|
| 终端用户 | 疗愈中的个人 | 主站（front） |
| 运营 / 调试 | 开发者、prompt 调优者 | 管理后台（admin，只读） |
| 系统 | 批处理任务、LLM | backend |

## 2. 设计原则

| ID | 原则 | 含义 |
|----|------|------|
| D1 | **陪伴优先于说教** | 先承接与合法化感受，再分析；避免「你应该」堆砌 |
| D2 | **深度与温暖并重** | 心理学温厚 + 生物学/社会学理性；回复随表达复杂度展开 |
| D3 | **长期记忆可解释** | 记忆来自画像、日摘要、事件锚点、近期对话；可审计、可批处理更新 |
| D4 | **隐私最小化（演进中）** | 当前 admin 可读消息体便于调试；中长期应收敛权限（见 `SECURITY_EVOLUTION.md`） |
| D5 | **三仓库、一套数据** | 前端薄客户端；智能与存储集中在 backend + Postgres |

疗愈伴侣 system prompt 的权威文案见 backend `src/chat/prompt.ts`；调试约定见 backend `docs/疗愈伴侣Prompt调试日志.md`（含 E1–E9）。

## 3. 范围（In / Out）

### 3.1 本期范围内（已实现）

- 用户注册 / 登录 / JWT 会话
- 连续对话（流式 SSE）+ 消息分页历史
- 停止生成并回滚最后一条用户消息
- 个人日记 CRUD（服务端异步向量化）
- 账户注销（级联删除用户数据）
- RAG：画像 + 近期对话 + 日摘要 + 锚点
- 夜间批处理：日摘要、画像更新、锚点维护
- 管理后台：用户列表与详情（只读）
- 生产部署：Docker Postgres + PM2 API + Nginx 静态站

### 3.2 明确不在本期

- 多模态（语音 / 图片）对话
- 多设备实时同步以外的离线优先 App
- 用户侧「设置页」（账号操作在页头菜单）
- 用户侧直接读写「画像 / 摘要 / 锚点」的 REST API
- 危机干预自动转接、强制上报
- 付费订阅与账单

## 4. 用户故事与验收要点

### 4.1 注册与登录

| 故事 | 验收 |
|------|------|
| 作为新用户，我可以用用户名+密码注册并进入对话 | 用户名 3–64、密码 6–128；成功返回 JWT；自动创建默认画像 |
| 作为老用户，我可以登录继续对话 | 校验密码；JWT 有效期约 7 天 |

### 4.2 疗愈对话

| 故事 | 验收 |
|------|------|
| 我可以倾诉并收到流式回复 | `POST /api/messages` 后 `POST /api/chat/stream`；前端逐字/逐块展示 |
| 长自述应得到更深入的回复 | 路由至 deep 模型（长度或深度信号触发）；篇幅随复杂度展开 |
| 短消息不必长篇大论 | fast 轨；仍保持温暖承接 |
| 我可以停止生成 | 中止 SSE；删除最后一条用户消息；草稿可恢复 |
| 向上滚动可加载更早消息 | `GET /api/messages?before=` 游标分页 |

### 4.3 日记

| 故事 | 验收 |
|------|------|
| 我可以写、改、删日记 | 标题+正文；列表分页；支持搜索（客户端批量拉取后过滤） |
| 日记进入长期记忆体系 | 写入/更新时异步 embedding（供批处理与后续扩展检索） |

### 4.4 账户

| 故事 | 验收 |
|------|------|
| 我可以注销账户 | `DELETE /api/account` 级联删除消息、日记、摘要、锚点、画像、用户 |

### 4.5 管理

| 故事 | 验收 |
|------|------|
| 运营可用密钥查看用户与用量 | `X-Admin-Key`；列表含消息/摘要/锚点计数；详情含画像与近期消息 |

## 5. 对话体验需求（质量）

主观标准（高于字数指标）：

1. **深入准确**：能把情节与机制说清，避免空洞鸡汤  
2. **情绪温暖且坚定**：合法化创伤激活，不贬低用户  
3. **接近优质网页对话**：以 DeepSeek 网页级回复为参照金样  

Prompt 改动纪律摘要：

- 优先全局软目标；确需补丁时用「当用户…时，你可以…」（场景清、动作松）
- 禁止编号 SOP、user 侧场景任务块、无场景硬约束（详见 Prompt 调试日志 E2/E3/E9）

## 6. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 对话首 token 可接受延迟；SSE 不被 Nginx 缓冲打断 |
| 可用性 | `/api/health` 可探活；PM2 守护 API |
| 安全 | 密码 bcrypt；JWT HS256；管理密钥与用户 JWT 分离；DB 不对公网暴露 |
| 可运维 | 三仓一键部署；`.env` 不进制品包；cron 可配置批处理 |
| 可观测 | 后端日志含模型与提示摘要；admin 可抽查对话（调试期） |

## 9. 风险与假设

| 风险 / 假设 | 应对 |
|-------------|------|
| LLM 输出不稳定 | Prompt 基线 + 调试日志；deep/fast 分流 |
| Embedding 依赖 OpenRouter | 对话可用 DeepSeek，但向量化仍需 OpenRouter Key |
| Admin 可读明文消息 | 调试便利 vs 隐私；按 `SECURITY_EVOLUTION.md` 演进 |
| 日记向量未进实时聊天 RAG | 当前日记主要用于批处理与扩展；需求若变更再接入 `match_diaries` |
